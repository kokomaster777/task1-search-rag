# Задача 1. Полнотекстовый vs векторный поиск + RAG

Стек: **Postgres 16** (FTS: русская морфология + pg_trgm) · **Qdrant** (эмбеддинги `intfloat/multilingual-e5-small`, локально) · **FastAPI** · **Ollama** `qwen2.5:3b-instruct` (текстовые ответы, локально, без API-ключей).

## Структура

```
task1-search-rag/
├── docker-compose.yml       # postgres + qdrant + backend
├── data/products.json       # каталог товаров (722 шт., сгенерирован)
├── ingest/generate_products.py
├── backend/
│   ├── Dockerfile           # CPU-torch, модель эмбеддингов прогрета в образ
│   ├── requirements.txt
│   └── app/
│       ├── main.py          # эндпоинты
│       ├── db.py            # Postgres FTS + триграммы (опечатки)
│       ├── vectors.py       # Qdrant + e5 (префиксы query:/passage:)
│       └── rag.py           # retrieval -> Ollama -> ответ
└── compare_queries.py       # демо-сравнение на подготовленных запросах
```

## Запуск

1. Установить [Ollama](https://ollama.com) на хост и скачать модель:

```bash
ollama pull qwen2.5:3b-instruct
```

2. Поднять сервисы:

```bash
docker compose up --build
```

При старте бэкенд сам загружает товары в обе базы (идемпотентно). Swagger: http://localhost:8000/docs
Бэкенд ходит к Ollama на хосте через `host.docker.internal:11434` (настраивается переменной `OLLAMA_URL`).

## Эндпоинты

| Метод | Путь | Что делает |
|---|---|---|
| GET | `/search/fulltext?q=...` | Полнотекстовый поиск Postgres |
| GET | `/search/vector?q=...` | Семантический поиск Qdrant |
| GET | `/search/compare?q=...` | Оба результата рядом |
| POST | `/ask` `{"question": "..."}` | RAG: эмбеддинг → топ-k → LLM → текстовый ответ |

## Результаты сравнения (compare_queries.py, 10 запросов)

- **Семантика выигрывает** на синонимах и перефразировках: «лэптоп для командировок», «техника, которая сама убирает квартиру» → находит нужное; FTS по таким запросам пуст (websearch-режим требует все слова запроса).
- **Полнотекст выигрывает** на точных вхождениях: бренды, граммовки («кофе Jardin 250 г») — точнее и дешевле (не нужен инференс модели). Вектор те же товары находит первыми, но «добирает» top-k похожим — не умеет отвечать «больше ничего нет».
- **Опечатки** («нотбук асус») ловят оба: в Postgres — за счёт триграммного fallback (pg_trgm), эмбеддинги устойчивы сами по себе.
- **Слабость обоих** — числовые конфигурации («16/512»): в проде решается фильтрами по структурированным атрибутам, а не текстовым поиском.
- Практический вывод: в проде — **гибрид** (FTS + вектор, слияние через RRF) + фильтры по атрибутам.

## Замечания

- Всё работает локально и бесплатно: эмбеддинги — sentence-transformers на CPU, генерация — Ollama.
- e5 требует префиксы `passage:` / `query:` — без них качество заметно падает.
- LLM проинструктирована отвечать только по переданным товарам (защита от галлюцинаций); качество формулировок ограничено размером модели (3B) — выбор обусловлен доступным железом (12 ГБ RAM), код совместим с любой моделью Ollama через `LLM_MODEL`.
- Данные синтетические: часть названий/цен выглядит неестественно (артефакты генератора), на выводы сравнения не влияет.
