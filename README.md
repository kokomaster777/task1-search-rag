# Задача 1. Полнотекстовый vs векторный поиск + RAG

Стек: **Postgres 16** (FTS: русская морфология + pg_trgm) · **Qdrant** (эмбеддинги `intfloat/multilingual-e5-small`) · **FastAPI** · **Claude API** (текстовый ответ).

## Структура

```
task1-search-rag/
├── docker-compose.yml       # postgres + qdrant + backend
├── data/products.json       # каталог товаров (722 шт., сгенерирован)
├── ingest/generate_products.py
├── backend/
│   ├── Dockerfile           # модель эмбеддингов прогрета в образ
│   ├── requirements.txt
│   └── app/
│       ├── main.py          # эндпоинты
│       ├── db.py            # Postgres FTS + триграммы (опечатки)
│       ├── vectors.py       # Qdrant + e5 (префиксы query:/passage:)
│       └── rag.py           # retrieval -> Claude -> ответ
└── compare_queries.py       # демо-сравнение на подготовленных запросах
```

## Запуск

```bash
export ANTHROPIC_API_KEY=sk-ant-...   # опционально; без ключа /ask вернёт только retrieval
docker compose up --build
```

При старте бэкенд сам загружает товары в обе базы (идемпотентно). Swagger: http://localhost:8000/docs

## Эндпоинты

| Метод | Путь | Что делает |
|---|---|---|
| GET | `/search/fulltext?q=...` | Полнотекстовый поиск Postgres |
| GET | `/search/vector?q=...` | Семантический поиск Qdrant |
| GET | `/search/compare?q=...` | Оба результата рядом |
| POST | `/ask` `{"question": "..."}` | RAG: эмбеддинг → топ-k → LLM → текстовый ответ |

## Что показывает сравнение (compare_queries.py)

- **Семантика выигрывает** на синонимах и перефразировках: «лэптоп для командировок» → ноутбуки; «техника, которая сама убирает» → робот-пылесос. FTS по таким запросам находит мало или ничего.
- **Полнотекст выигрывает** на точных вхождениях: бренды, артикулы, «кофе Jardin 250 г» — точнее и дешевле (не нужен инференс модели).
- **Опечатки**: pg_trgm ловит «нотбук асус», эмбеддинги тоже устойчивы к ним.
- Практический вывод: в проде обычно берут **гибрид** (FTS + вектор, слияние через RRF) — лучшие качества обоих.

## Замечания

- Эмбеддинги локальные и бесплатные (CPU достаточно для 722 товаров, индексация ~10-20 сек).
- e5 требует префиксы `passage:` / `query:` — без них качество заметно падает.
- LLM-слой: `claude-haiku-4-5`, контекст — только найденные товары, модель проинструктирована не выдумывать.
